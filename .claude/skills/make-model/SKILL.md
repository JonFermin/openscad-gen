# make-model: OpenSCAD + 3D Preview Generator

## Flags

- **`--verbose`** (default: **on**) — Narrate each self-evaluation pass: show the screenshot, explain what looks wrong, and describe the fix before re-generating. Pass `--quiet` to suppress iteration narration and only present the final result.

---

Produce two deliverables from a description of a physical object:

1. **`<name>.scad`** — A parametric OpenSCAD file ready for rendering/export/printing
2. **`<name>_manifest.json`** — A scene manifest JSON for the Three.js previewer

Both files represent the same object. The OpenSCAD file is the source of truth for manufacturing;
the Three.js preview is a fast visual approximation.

---

## Phase 1: Clarifying Questions (1 round max)

Before generating, gather enough information. Ask only what's missing — if the user's description
is already detailed, skip to generation. Default aggressively.

### Always needed (ask if missing):

1. **Overall dimensions** — "Roughly how big? length x width x height, or diameter x height. Units — mm or inches?"
2. **Purpose** — Affects wall thickness, tolerances, detail level:
   - 3D printing (FDM/resin)
   - Laser cutting
   - Visualization only
   - CNC / machining
   - Just exploring the shape

### Ask if relevant:

3. **Parametric controls** — "Which dimensions should be easy to change?"
4. **Fillet/chamfer** — "Sharp edges or rounded?"
5. **Wall thickness** — Default to 2mm for 3D printing if not specified.
6. **Tolerances** — Default to 0.2mm clearance for FDM.
7. **Mounting / attachment** — Screw holes, snap fits, slots, tabs?

### Don't ask about:
- Color, material, file format beyond .scad

Keep to 1 round max. A model with reasonable defaults that the user can tweak beats an interview.

## Phase 2: Generate the OpenSCAD File

**First, read `references/openscad-guide.md`** for syntax patterns and best practices.

### Structure

Every .scad file follows this layout:

```
// ============================================
// <Object Name>
// Generated by make-model
// Units: <mm|in>
// ============================================

// --- Parameters ---
// (all tunable dimensions as variables at the top)

// --- Modules ---
// (reusable sub-shapes as module definitions)

// --- Assembly ---
// (final object composed from modules)
```

### Key principles:

- **Parameters at the top.** Every dimension the user might want to change is a named variable.
- **Modules for sub-shapes.** Any repeated or logically distinct part gets its own `module`.
- **CSG thinking.** Build complex shapes via union(), difference(), intersection().
- **$fn for curves.** Set `$fn` as a parameter (default 64 for preview, 128 for final).
- **Print-ready defaults** for 3D printing:
  - Minimum wall thickness: 2mm
  - Minimum detail: 0.4mm (FDM nozzle width)
  - Add 0.1mm to holes for tolerance
  - Avoid unsupported overhangs > 45°
  - Manifold geometry (no open edges, no zero-thickness walls)
- **z-fighting avoidance:** Extend cutting shapes 0.5–1mm beyond surfaces in difference() ops.
- **Comments explain why**, not what.

### Output

Save to `./output/<name>.scad` with a descriptive filename.

## Phase 3: Generate the Scene Manifest JSON

Create a `./output/<name>_manifest.json` that describes the Three.js geometry to render.

### Manifest format:

```json
{
  "name": "Object Name",
  "units": "mm",
  "parts": [
    {
      "id": "base",
      "label": "Base Plate",
      "type": "box",
      "params": { "width": 50, "height": 5, "depth": 30 },
      "position": [0, 2.5, 0],
      "rotation": [0, 0, 0],
      "color": "#4488cc",
      "opacity": 1.0
    },
    {
      "id": "hole1",
      "label": "Mounting Hole",
      "type": "cylinder",
      "params": { "radiusTop": 2, "radiusBottom": 2, "height": 7 },
      "position": [10, 2.5, 8],
      "rotation": [0, 0, 0],
      "color": "#ff4444",
      "opacity": 0.3,
      "wireframe": true,
      "operation": "difference"
    }
  ],
  "camera": {
    "distance": 120,
    "angle": [30, 45]
  }
}
```

### Geometry types:

| type | params |
|------|--------|
| `box` | `width`, `height`, `depth` |
| `sphere` | `radius` |
| `cylinder` | `radiusTop`, `radiusBottom`, `height`, `segments` (optional) |
| `extrude` | `shape` (array of [x,y] points), `height` |
| `lathe` | `points` (array of [x,y] profile points), `segments` |

### Coordinate system:

Three.js uses Y-up. When translating from OpenSCAD (Z-up):
- OpenSCAD X → Three.js X
- OpenSCAD Y → Three.js Z (negated)
- OpenSCAD Z → Three.js Y

### difference() visualization:

For subtracted volumes, include them in the manifest with:
- `"operation": "difference"`
- `"wireframe": true`
- `"color": "#ff4444"`
- `"opacity": 0.3`

This renders them as transparent red wireframe overlays so the user can see what's being removed.

## Phase 4: Review & Fix Loop (Agent Team)

After generating both files, run a review → fix loop using two specialized agents.
This replaces self-evaluation with independent agents that have fresh context.

### Step 1: Ensure previewer is running

If not already started in this session, run in the background:

```bash
cd previewer && npm run dev
```

The previewer persists across iterations — no need to restart.

### Step 2: Spawn the reviewer agent

Use the Agent tool with `subagent_type: "review-model"` and provide:

```
Review the model preview against this spec.

**Original spec:** <paste the user's original description/requirements>

**Manifest file:** output/<name>_manifest.json

**Manifest content:**
<paste the full manifest JSON>
```

The reviewer will:
1. Analyze the manifest for red flags (inverted rotations, bad Y-positions, missing parts)
2. Navigate to the preview and take a screenshot
3. Evaluate against a 6-point checklist (orientation, shape, proportions, completeness, positioning, plausibility)
4. Return a structured PASS/FAIL verdict with specific issues and suggested fixes

### Step 3: If PASS → proceed to Phase 5

### Step 4: If FAIL → spawn the fixer agent

Use the Agent tool with `subagent_type: "fix-model"` and provide:

```
Fix the following issues found by the reviewer.

**Reviewer's issues:**
<paste the reviewer's full issues list and suggested fixes>

**Manifest file:** output/<name>_manifest.json
**Source file:** output/<name>.scad
```

The fixer will:
1. Read the manifest and locate the broken parts
2. Apply targeted, surgical edits (not a full rewrite)
3. Return a summary of exactly what changed and why

### Step 5: Re-review after fix

After the fixer completes, spawn the reviewer agent again with the updated manifest.
This ensures the fix actually worked and didn't introduce new issues.

### Loop control

- Maximum **3 review cycles** (review → fix → re-review counts as one cycle)
- If still failing after 3 cycles, present the result to the user with a note about
  what the reviewer is still flagging so they can decide how to proceed.

### Troubleshooting:
- **Screenshot shows empty state ("No manifest loaded"):** Verify the manifest file exists and the filename in the URL matches exactly.
- **Connection refused:** The previewer isn't running. Start it and retry.

---

## Phase 5: Present to User

After review passes, present the result:

1. **What you built** — One sentence.
2. **Preview** — Note that a 3D preview is running at localhost:3000.
3. **Key parameters** — Top 3–5 parameters the user is most likely to tweak.
4. **Simplifications** — Any CSG operations approximated in the preview.
5. **Print notes** — If 3D printing: overhangs, thin walls, orientation suggestions.

To iterate: the user can ask to change dimensions or add features. Regenerate both files on each
iteration — they must stay in sync. Re-run the screenshot self-evaluation loop after each change.
